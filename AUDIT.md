# AUDIT.md — Fable-5 audit of the Sonnet-authored portfolio plans

**What this file is.** The planning task for these 12 projects was run by Sonnet instead of Fable 5. Sonnet produced correct *scaffolding* (12 folders, each with `PLAN.md` / `PROFESSOR-NOTES.md` / `RESOURCES.md`, plus `ROADMAP.md`) at Sonnet-level *depth*. This document is the record of the diff: for every project, what Sonnet got right, what was shallow/generic/wrong/missing, and what the Fable-5 revision changed. Resource-path claims were re-verified against fresh clones of all 19 upstream repos (not trusted from Sonnet's self-reported ✅ flags). Structural decisions (keep/merge/cut, new projects) and their reasoning are recorded here — nothing was changed silently.

**Verdict up front.** This was *not* a case of shallow stubs. Sonnet's baseline is genuinely above average: every PLAN has a quantified success-criteria table and an explicit failure-mode section; every PROFESSOR-NOTES uses a real what/why/how/where structure with comprehension questions; every RESOURCES carries dated status flags and honestly discloses 404s. The revision is therefore **surgical, not a demolition** — it credits what stands and deepens what an implementer would still have to re-derive. The gap between Sonnet-level and the standard the spec demanded ("all hard thinking must live in the plans, not be deferred to implementation time") is concentrated in nine systemic areas.

---

## Systemic gaps (apply to all 12 projects)

1. **RESOURCES.md defers the hard work.** Every LangGraph and Agno citation resolves to "this path 404s — clone the repo and `find` it yourself at implementation time." That is exactly the deferral the spec forbids. **Fixed** by resolving the actual current paths against real clones (see "Path verification" below) and citing exact files.
2. **No prompt-engineering specs anywhere.** Plans say "the supervisor is a structured-output LLM call" or "the judge scores against a rubric" but never give the system-prompt skeleton, the rubric text, the grader prompt, or a few-shot strategy. Prompts are the highest-variance part of any agent build; leaving them blank forces the implementer to re-invent them. **Fixed** by adding a prompt/interface appendix to each PLAN.
3. **Undefined interfaces and contracts.** Pydantic sub-models are referenced by name but never defined (`MarketDataResult`, `SentimentResult`, `CritiqueVerdict`, `ValidationVerdict`, per-doc-type extraction schemas…); tool functions are named but not typed (args / returns / error semantics); no FastAPI request/response shapes; no DB DDL for the `audit_log`, `idempotency_keys`, or scorecard tables. **Fixed** with concrete type/DDL sketches (pseudocode, no implementation).
4. **No systematic error taxonomies.** Failure handling is mentioned ad hoc for one or two tools per project (yfinance empties in P01) but never as a per-tool matrix of {timeout, rate-limit, empty, malformed} → {what state gets written, retry policy}. **Fixed** with an error-handling table per project.
5. **No model-selection guidance.** Cost is given as a total ($0.25/report) but never derived, and no plan says which model *tier* to use per role (a cheap model for routing, a strong model for the judge), what temperature, or what the context budget is. **Fixed** with a per-role model table.
6. **"Done" definitions are good but test plans are hand-waved.** "Unit-tested with a mocked LLM" is asserted without saying what the tests assert. **Fixed** by naming the concrete tests per phase.
7. **PROFESSOR-NOTES stop one level short of first principles.** They explain *what* structured output is, not *how* constrained decoding / tool-call APIs actually enforce it; *that* embeddings exist, not what cosine similarity computes or why chunk size matters. Prerequisites cite "any short primer" rather than the specific resource the spec asked for. **Fixed** by teaching the mechanism and citing named resources.
8. **Comprehension questions have no answer keys.** A student can't self-check. **Fixed** by adding an answer key after each question set.
9. **Cross-project contracts are never pinned.** P03/P11/P12 all need to observe a target agent's internal tool calls, but P03 simultaneously insists the target is a "black box over HTTP" — a genuine architectural contradiction. P12 depends on P03's regression math, which P03 never specifies. **Fixed** by defining two shared contracts once (below) and referencing them everywhere.

### Two shared contracts introduced by the revision

- **Target Agent Contract** (emitted by 01/02, consumed by 03/11/12/13): a compliant agent exposes (a) an HTTP invoke endpoint; (b) in its response, a structured *trajectory* — the ordered list of tool calls with args, results, token counts, and latencies — **or** an OpenTelemetry GenAI export of the same; (c) a `version` string. This resolves the black-box-vs-introspection contradiction: the target stays black-box in that the harness never imports its internals, but it is *observable* because the contract requires it to emit its own trajectory. New **Project 13** is the reference implementation of the tracing half of this contract.
- **MCP Server Contract** (04/08/09): stdio transport for local dev, **Streamable HTTP** for remote (not the deprecated HTTP+SSE Sonnet cites); Inspector-clean; typed tool/resource/prompt schemas.

## Per-project findings

Each entry: **Right** (specific credit) → **Gaps** (what the revision deepens) → the resource fixes already covered above.

### 01 · Multi-Agent Financial Research Analyst
**Right.** Strong measurable metric table; a real mermaid diagram; an agent roster with per-agent state reads/writes; a `ResearchState` TypedDict; the supervisor pattern is correctly justified over a ReAct loop; the risks section is genuinely sophisticated (yfinance silent empties, re-running only the Report Generator on critic revision to control cost, section-aware chunking). Tech-stack table has a rejected-alternative column.
**Gaps.** The sub-result models (`MarketDataResult`, `SentimentResult`, `FundamentalsResult`, `QuantResult`, `CritiqueVerdict`) are named but never defined — the implementer designs the core data model from scratch. No prompts: the supervisor routing prompt, the CRAG grader prompt (and the unmade decision of binary vs. 1–5 grading), the critic rubric text, and the report citation format are all absent — yet the citation-integrity checker in §6 *cannot be built* without the citation format. The quant sandbox says "restricted exec namespace" without choosing a mechanism (RestrictedPython vs. subprocess-with-rlimits vs. curated-globals `exec`) — security-sensitive and must not be left open. yfinance rate-limit mitigation is named but has no strategy. The 15 eval tickers, the judge model, and the rubric anchors are unspecified. No embedding model, chunk size, or overlap is chosen. PROFESSOR-NOTES should teach *how* structured output is enforced and *why* a structured-output router beats a ReAct loop (error compounding across turns), and needs an answer key.

### 02 · Document-Processing HITL Pipeline
**Right.** Excellent tech rationale (Postgres-not-Redis for the system-of-record; a deterministic rules engine rather than an LLM validator); `interrupt()` durable-execution semantics explained correctly; idempotency keyed on content-hash + action-type; the "kill the process mid-approval and resume" test; the confidence≠risk distinction; per-doc-type escalation timeouts.
**Gaps.** The `extracted_fields` discriminated union is named but the per-doc-type schemas (invoice vs. claim vs. contract) are never defined — again the core data model is left to the implementer. The business rules are never enumerated beyond one "amount > $10k" example — this project needs a starter rule catalog per type with thresholds. No DDL for `audit_log` or the idempotency table, and the atomic unique-constraint pattern is only alluded to in a PROF question, not specced. The Slack Block Kit approval payload and the `thread_id ↔ document_id` resume mapping are unspecified. **Biggest hole:** `extraction_confidence` is consumed everywhere but its *source* is never stated — LLM self-reported confidence is a known trap; the plan must pin a real mechanism (logprob-derived, self-consistency voting, or a separate verifier). Synthetic-data generation (the prompt that emits document + ground-truth JSON, plus diversity controls and a cost estimate) is missing. OCR is punted as "optional" without a recommendation.

### 03 · Agent Evaluation & Observability Platform
**Right.** The "rarest project" positioning is correct and valuable; judge calibration against hand-labels with Cohen's κ ≥ 0.6 and a 3/3 injected-regression test is genuinely strong design; the black-box-target principle; the scripted-vs-adversarial split; scoring process (`tool_call_correctness`) separately from outcome (`task_success`).
**Gaps.** The rubric is never written (dimensions named, anchors absent) and the judge prompt isn't specced. The 15 scenarios (10 scripted + 5 adversarial) — the actual hard design work — are entirely deferred. **Regression math is unspecified:** "statistics (no LLM)" with no threshold, and with n≈30 runs the plan never discusses variance or the risk of a flaky gate (needs a two-proportion test or bootstrap CI, not a bare Δ). **Architectural contradiction:** the harness is meant to capture "every tool call" while treating the target as a black box over HTTP — impossible unless the target *emits* its trajectory. This is resolved by the new **Target Agent Contract** (and Project 13). κ on 20 binary labels is statistically fragile — the calibration sample needs stratification or a larger n. Add answer keys and teach why κ (chance-corrected) beats raw agreement, plus the standard judge biases (position, verbosity, self-enhancement).

### 04 · MCP-Native Personal Ops Agent
**Right.** The provider-vs-consumer framing is exactly the right emphasis; Inspector as a hard compliance gate; the stub-before-OAuth sequencing; an *absolute* approval gate with structural (not prompt-level) enforcement; the deliberate tools-vs-resources distinction.
**Gaps.** The three servers' tool schemas are named but not typed, and `find_free_slots` hides a real algorithm (interval subtraction over a working-hours config) that goes unmentioned. Memory is shallow: no write policy (when does the agent decide to store a memory?), no consolidation/dedup for contradictory preferences over time, no supersede/decay for "I changed my mind." The proactive digest's relevance logic is unspecified. The approval gate's structural enforcement is stated as a goal but the mechanism (an execute-fn that is unreachable without a decision object) isn't sketched. **Stale protocol detail:** "HTTP+SSE" was replaced by Streamable HTTP in the 2025 spec revision — fix and record. The 2-week wall-clock trial should offer a compressed-simulation alternative for the DoD.

### 05 · Self-Healing SQL Analytics Agent
**Right.** An honest ≥70% target with a "report it even if ugly" note; execution-accuracy (not string-match) as the metric; the read-only boundary as *structural* safety with an adversarial DoD test; explicit self-awareness that this is the simplest orchestration; caching schema per DB; reusing Spider's own eval script.
**Gaps.** **The read-only mechanism is described in Postgres vocabulary ("read-only DB role") but Spider databases are SQLite, which has no roles** — the real mechanism is `mode=ro` connection URI or `PRAGMA query_only`. Sonnet's "SQLite or Postgres with a read-only role" glosses a genuine gotcha; both mechanisms must be specced. The 100 questions / 3–5 DBs are never pinned, nor is the selection method (stratify by Spider's easy/medium/hard/extra tags). The self-check rules ("absurd values," "plausibly empty") aren't enumerated as a concrete checklist with thresholds, and the `prior_feedback` retry-prompt template isn't given. A cheap pre-execution `sqlglot` parse + statement-type allowlist (defense in depth + better feedback) is worth adding. The chart-spec schema the Narrative agent emits is undefined.

### 06 · Codebase Onboarding Agent
**Right.** The run-it-live-on-the-interviewer's-repo pitch; 100% code-checked citations; static import graph rather than LLM-inferred architecture; excluding vendored/generated dirs; the held-out-repo robustness test; the Python-`ast` fallback scoping.

## Portfolio-level analysis: keep / merge / cut / add

**No cuts, no merges.** The 12 projects are distinct enough to each earn their place. The two apparent overlaps are intentional layering and are *kept*, with the boundary sharpened in both plans:
- **03 ↔ 12:** 03 builds the eval harness; 12 packages that harness as an installable CI product. Different skills (eval design vs. release engineering). Keep both.
- **04 ↔ 09:** 04 *consumes* MCP servers (plus authors a couple as a means to an end); 09 is *pure authoring + publishing*. Keep both; 04's plan now explicitly points forward to 09 for the deep authoring story.

**Gap analysis vs. 2026 hiring value.** The existing set covers multi-agent orchestration (01, 08), HITL/durable execution (02), evals (03), MCP consume+author (04, 09), text-to-SQL (05), code RAG (06), security (07), RAG depth (10), guardrails (11), and CI (12). Genuinely *uncovered* high-value areas: **runtime observability/tracing** (as opposed to 03's black-box quality eval), **long-term agent memory as a first-class system**, and **computer-use / browser automation**. Voice/realtime and deep-research were considered and dropped — voice is a weaker signal for this candidate's multi-agent focus, and deep-research overlaps too much with 01's research-synthesis pattern to justify a slot.

**Three new projects added (numbered 13–15):**

- **13 · Agent Observability Stack.** OpenTelemetry GenAI semantic conventions + span/trace instrumentation of agent runs, a trace viewer, and cost/latency/error dashboards. Distinct from 03: 03 *judges output quality* as a black box; 13 *instruments the runtime internals*. Crucially, 13 is the reference implementation of the **Target Agent Contract**, so it retro-justifies and unblocks 03/11/12 (all three need to observe a target's internal steps). Highest-leverage addition.
- **14 · Long-Term Memory System.** Episodic / semantic / procedural memory with explicit write → consolidate → forget/supersede policies and retrieval, benchmarked on a multi-session recall task. 04 has memory-lite as a feature; 14 makes memory the subject, which is where the 2026 "stateful agents" hiring signal actually lives.
- **15 · Computer-Use / Browser Automation Agent.** A Playwright-driven agent combining DOM and vision, with task-success evaluation on a fixed web-task set and a safety/confirmation gate on destructive actions. Absent from the portfolio today, the strongest live demo, and one of the hottest 2026 capabilities.

Each new project's own PLAN.md repeats its "what this adds beyond the current set" justification in context.

**ROADMAP consequence.** The final set is 15 projects. The rewritten ROADMAP re-sequences around the two shared contracts (13 becomes an early enabler, not an afterthought), adds an explicit "lead with these on your resume" ranking (the spec asked for it and Sonnet only had a fast-track), and surfaces the shared-contract section so the inter-project dependencies are legible.
